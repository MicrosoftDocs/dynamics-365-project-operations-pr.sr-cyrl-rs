# <a name="upgrade-from-project-service-automation-to-project-operations"></a>Надоградња са апликације Project Service Automation на услугу Project Operations

Са узбуђењем најављујемо другу од три фазе надоградње са решења Microsoft Dynamics 365 Project Service Automation на Microsoft Dynamics 365 Project Operations. Овај чланак пружа преглед за клијенте који крећу на ово узбудљиво путовање. 

Програм испоруке надоградње биће подељен у три фазе.

| Испорука надоградње | 1. фаза (јануар 2022.) | 2. фаза (новембар 2022.) | 3. фаза (априлски талас 2023.)  |
|------------------|------------------------|---------------------------|---------------------------|
| Нема зависности на структурној анализи посла (САП) за пројекте | :heavy_check_mark: | :heavy_check_mark: | :heavy_check_mark: |
| САП унутар тренутно подржаних ограничења за Project Operations | | :heavy_check_mark: | :heavy_check_mark: |
| САП изван тренутно подржаних ограничења за Project Operations, укључујући подршку за Project Desktop Client | | | :heavy_check_mark: |

## <a name="upgrade-process-features"></a>Функције процеса надоградње 

У склопу процеса надоградње додали смо евиденције надоградње на мапу локације како бисмо омогућили администраторима да лакше дијагностикују неуспехе. Поред новог интерфејса, биће додата и нова правила за проверу исправности како би се осигурао интегритет података након надоградње. Следеће провере исправности биће додате процесу надоградње.

| Валидације | 1. фаза (јануар 2022.) | 2. фаза (новембар 2022.) | Фаза 3  |
|-------------|------------------------|---------------------------|---------------------------|
| САП ће бити валидиран према уобичајеним кршењима интегритета података (на пример, доделе ресурса које су повезане са истим надређеним задатком, али имају различите надређене пројекте). | | :heavy_check_mark: | :heavy_check_mark: |
| САП ће бити валидиран према [познатим ограничењима за Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries). | | :heavy_check_mark: | :heavy_check_mark: |
| САП ће бити валидиран према познатим ограничењима за Project Desktop Client. | |  | :heavy_check_mark: |
| Књиговодствени ресурси и календари пројекта биће процењени у односу на уобичајене некомпатибилне изузетке правила календара. | | :heavy_check_mark: | :heavy_check_mark: |

У фази 2, клијенти који надограде на Project Operations имаће своје постојеће пројекте надограђене на искуство само за читање за планирање пројекта. У овом искуству које је само за читање, цео САП биће видљив у матрици координатне мреже за праћење. Да би уредили САП, пројектни менаџери могу да изаберу [**Конвертуј**](/PSA-Upgrade-Project-Conversion.md) на главној страници пројекта. Процес у позадини затим ажурира пројекат тако да подржава ново искуство заказивања пројекта из решења Project for the Web. Ова фаза је прикладна за клијенте који имају пројекте који се уклапају у [позната ограничења решења Project for the Web](/project-for-the-web/project-for-the-web-limits-and-boundaries).

У 3. фази, биће додата подршка за Project desktop client, у корист клијената који желе да наставе да уређују своје пројекте из те апликације. Међутим, ако постојећи пројекти буду конвертовани у ново искуство решења Project for the Web, приступ програмском додатку биће онемогућен за сваки конвертовани пројекат.

## <a name="prerequisites"></a>Предуслови

Да бисте испунили услове за надоградњу 1. фазе, морате да испуните следеће критеријуме:

- Циљно окружење не сме да садржи записе у **msdyn_projecttask** ентитету.
- Важеће лиценце за Project Operations морају бити додељене свим активним корисницима. 
- Морате да проверите ваљаност процеса надоградње у најмање једном непроизводном окружењу које садржи репрезентативни скуп података који је усклађен са производним окружењем.
- Циљно окружење мора бити ажурирано на 37 верзију ажурирања решења Project Service Automation (V3.10.58.120) или новију верзију.

Да бисте испунили услове за надоградњу 2. фазе, морате да испуните следеће критеријуме:

- Важеће лиценце за Project Operations морају бити додељене свим активним корисницима. 
- Морате да проверите ваљаност процеса надоградње у најмање једном непроизводном окружењу које садржи репрезентативни скуп података који је усклађен са производним окружењем.
- Циљно окружење мора бити ажурирано на 37 верзију ажурирања решења Project Service Automation (V3.10.58.120) или новију верзију.
- Окружења која садрже задатке (msdyn_projecttask) подржана су само ако је укупан број задатака по пројекту 500 или мањи.

Предуслови за 3. фазу биће ажурирани како се приближава датум опште доступности.

## <a name="licensing"></a>Лиценцирање

Ако имате активне лиценце за Project Service Automation, можете инсталирати и користити решење Project Operations, што укључује све могућности решења Project Service Automation и још много тога. На тај начин можете да тестирате могућности решења Project Operations док настављате да користите Project Service Automation у производњи. Када лиценце за Project Service Automation истекну, мораћете да пређете на Project Operations. Када планирате овај прелаз, морате да рачунате на чињеницу да лиценца за Project Operations не укључује лиценцу за Project Service Automation. Клијенти који имају сценарије у којима су применили Project Service Automation и морају да наставе да користе или повећавају своје лиценце за PSA док намеравају да пређу на Project Operations могу да затраже привремене PSA лиценце на основу купљених лиценци за Project Operations. Једна Project Service Automation лиценца биће издата за једну Project Operations лиценцу. Привремене PSA лиценце могу бити затражене путем овог линка: aka.ms/ineedpsa

## <a name="testing-and-refactoring-customizations"></a>Прилагођавања тестирања и рефакторинга

Као полазну тачку, увезите сва прилагођавања у чисто Project Operations (једноставна примена) окружење како бисте потврдили да је увоз успешан и да се пословне операције понашају као што је очекивано.

Ево неких ствари на које треба да обратите пажњу:

- Увоз можда неће успети због зависности које недостају. Другим речима, референтна поља прилагођавања или друге компоненте које су уклоњене у услузи Project Operations. У овом случају, уклоните ове зависности из развојног окружења.
- Ако ненадгледано и комплетно решење укључују компоненте које нису прилагођене, уклоните те компоненте из решења. На пример, када прилагодите ентитет **Пројекат**, додајте само заглавље ентитета у решење. Немојте додавати сва поља. Ако сте претходно додали све поткомпоненте, можда ћете морати ручно да креирате ново решење и додате му релевантне компоненте.
- Обрасци и прикази се можда неће приказивати на очекивани начин. У неким околностима, ако сте прилагодили неки од готових образаца или приказа, прилагођавања могу да спрече ступање на снагу нових исправки у услузи Project Operations. Да бисте идентификовали ове проблеме, препоручујемо да урадите компаративни преглед чисте инсталације решења Project Operations и инсталације решења Project Operations која укључује прилагођавања. Упоредите најчешће коришћене обрасце у пословању да бисте потврдили да ваша верзија обрасца и даље има смисла и да вам не недостаје нешто из чисте верзије обрасца. Урадите исти тип компаративног прегледа за све приказе које сте прилагодили.
- Пословна логика може отказати при времену извршавања. Пошто референце на поља у додатним компонентама нису потврђене у тренутку увоза, пословна логика може да не успе због референци на поља која више не постоје и можда ћете добити поруку о грешци која је слична следећем примеру: „Ентитет 'Project' не садржи атрибут са Name = 'msdyn_plannedhours' и NameMapping = 'Logical'“. У том случају, измените прилагођавања тако да користе нова поља. Ако користите аутоматски генерисане прокси класе и референце јаког типа у логици додатне компоненте, размислите о поновном генерисању тих проксија из чисте инсталације. На овај начин можете лако да идентификујете сва места на којима додатне компоненте зависе од застарелих поља.

Када ажурирате прилагођавања да бисте чисто увезли решење Project Operations, пређите на следеће кораке.

## <a name="end-to-end-testing-in-development-environments"></a>Комплетно тестирање у развојним окружењима

### <a name="initiate-upgrade"></a>Започни надоградњу 

1. У Power Platform центру администрације, пронађите и изаберите ваше окружење. Затим, у апликацијама, пронађите и изаберите **Dynamics 365 Project Operations**.
2. Изаберите **Инсталирај** за почетак надоградње. Power Platform центар администрације ће представити ову инсталацију као нову инсталацију. Међутим, биће детектовано присуство претходне верзије решења Project Service Automation и постојећа инсталација ће бити надограђена.

    Када се надоградња доврши, окружење би требало да покаже да је Project Operations инсталирано, а да Project Service Automation није инсталиран.

    У зависности од количине података у окружењу, за надоградњу може бити потребно неколико часова. Основни тим који управља надоградњом треба да планира у складу са тим и да покрене надоградњу ван радног времена. У неким случајевима, ако је количина података велика, надоградња би требало да се покрене током викенда. Одлука о планирају треба да буде заснована на резултатима тестирања у нижим окружењима.

3. Надоградите прилагођена решења на одговарајући начин. У овом тренутку, примените све промене које сте направили у прилагођавањима у одељку [Прилагођавања тестирања и рефакторинга](#testing-and-refactoring-customizations) овог чланка.
4. Идите на **make.powerapps.com**, са падајућег менија у горњем десном углу портала изаберите ваше окружење, изаберите **Решења** из левог менија, изаберите решење **Застареле компоненте решења Project Operations** и **Деинсталирајте**.

    Ово решење је привремено решење које садржи постојећи модел података и компоненте које су присутне током надоградње. Уклањањем овог решења уклањате сва поља и компоненте које се више не користе. На тај начин ћете поједноставити интерфејс и олакшати интеграцију и проширење.
    
### <a name="upgrade-to-project-operations-lite"></a>Надоградња на Project Operations једноставну примену

Следећи кораци описују процес надоградње и повезано евидентирање грешака:

1. **Провера PSA верзије:** Да бисте инсталирали Project Operations, морате да имате V3.10.58.120 или новију верзију.
1. **Претходна провера ваљаности:** Када администратор иницира надоградњу, систем покреће операцију претходне провере ваљаности за сваки ентитет који представља језгро решења Project Operations. Овај корак потврђује да су референце свих ентитета важеће и обезбеђује да подаци који су повезани са САП-ом буду у оквиру објављених граница решења Project for the Web.
1. **Надоградња метаподатака:** Након успешне претходне провере ваљаности, систем покреће промене шеме и креира решење за застареле компоненте. Ово застарело решење можете уклонити након што довршите сав потребан рефакторинг и прилагођавања. Овај корак је најдужи део процеса надоградње и може да потраје до четири сата.
1. **Надоградња података:** Када се све потребне промене шеме доврше у кораку надоградње метаподатака, подаци се мигрирају у нову шему и раде се сва потребна подешавања подразумеваних вредности и поновно израчунавање.
1. **Ажурирање механизма за распоред пројекта:** Након успешне надоградње података, картица **Распоред** на главној страници је поново означена као **Задаци**. Када корисник изабере ову картицу након надоградње, усмерава се да оде до матрице координатне мреже за праћење да би приказао верзију САП-а која је само за читање. Да би уредио САП, мора да започне [процес конверзије](/PSA-Upgrade-Project-Conversion.md) распореда. Сви пројекти без постојећег САП-а могу директно да користе ново искуство планирања, без конверзије.
 
### <a name="validate-common-scenarios"></a>Провера уобичајених сценарија

Када проверавате ваљаност одређених прилагођавања, препоручујемо да прегледате и пословне процесе који су подржани у свим апликацијама. Ови пословни процеси укључују, али се не ограничавају на, креирање ентитета продаје као што су понуде и уговори, као и креирање пројеката који укључују САП-ове и одобрење стварних вредности.

## <a name="major-changes-between-project-service-automation-and-project-operations"></a>Главне промене између решења Project Service Automation и Project Operations

Овај одељак садржи резиме великих промена које можете очекивати између решења Project Service Automation и Project Operations.

### <a name="project-planning"></a>Планирање пројекта

Могућности планирања пројекта у Project Operations се више не ослањају на комбинацију логике на страни клијента и логике на страни сервера. Уместо тога, Project Operations користи Project for the Web као механизам за планирање. Ова промена могућности планирања омогућава неколико нових функција, као што су прикази табле и Гантовог дијаграма, планирање засновано на ресурсима, [ставке контролне листе задатака](https://support.microsoft.com/office/use-task-checklists-in-microsoft-project-for-the-web-c69bcf73-5c75-4ad3-9893-6d6f92360e9c) и режими планирања пројекта. Нове могућности планирања такође подржавају богат скуп нових [интерфејса за програмирање апликација (API)](../project-management/schedule-api-preview.md). Ови API-ји су намењени да обезбеде да ниједна програмска операција за креирање, ажурирање или брисање ентитета у САПУ-у не оштети израчуната поља у распореду.

У оквиру континуираних улагања у Project Operations, у Наплати и одређивању цена доступно је неколико нових могућности. У наставку су наведени неки примери:

- [Управљање подуговорима](../pro/subcontracting/managing-subcontracts-overview.md)
- [Аванси и уговори засновани на авансним уплатама](../pro/sales/set-up-advances-retainer-based-contracts-sales.md)
- [Статусом и провере уговора које не смете премашити](../pro/proforma-invoicing/manage-nte-status-validations-sales.md)
- Наплата заснована на задатку

### <a name="resource-management"></a>Управљање ресурсима

Project Operations пружа опциону подршку за Universal Resource Scheduling (URS) таблу и помоћника за заказивање. Ово ново искуство постаће обавезно у таласу издања за април 2023.

### <a name="how-can-i-install-project-operations-before-the-upgrade-tooling-is-available"></a>Како могу да инсталирам Project Operations пре него што алатке за надоградњу буду доступне?

Постоје две опције за инсталирање решења Project Operations пре него што алатка за надоградњу буде доступна:

- Обезбедите ново окружење.
- Примените Project Operations засебно на било коју продајну организацију на којој Project Service Automation није присутан.

Ако је Project Service Automation инсталиран у организацији, али се не користи, можете га деинсталирати. Када у потпуности уклоните Project Service Automation, Project Operations можете инсталирати у истој организацији.
